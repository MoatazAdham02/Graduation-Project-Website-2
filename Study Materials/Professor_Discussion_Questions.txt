2. MEDICAL / CLINICAL DOMAIN
--------------------------------------------------------------------------------

Q: What is DICOM and why is it important?
A: DICOM (Digital Imaging and Communications in Medicine) is the international
   standard for medical imaging. It defines how images and metadata (patient ID,
   study date, modality) are stored and exchanged. Your app accepts .dcm/.dicom
   and extracts metadata (e.g. patient name, study date, modality) for display.

Q: What modalities does COROnet support?
A: Echo (echocardiography), Cardiac CT, Cardiac MRI – as reflected in the dashboard
   and recent activity (e.g. "Echo", "Cardiac CT", "Cardiac MRI").

Q: How do you ensure medical data is handled responsibly?
A: [Tie to your Study Materials: DICOM_Data_Confidentiality.txt – confidentiality,
   access control, secure storage. Mention: only authorized users (JWT), secure
   backend, optional encryption at rest/transit if you implemented it.]

Q: Can a doctor rely on the "AI" results in production?
A: [Prepare: your project demonstrates workflow and integration; any AI/accuracy
   metrics (e.g. "98%") are illustrative or from literature. In real deployment,
   validation, regulatory approval, and clinical responsibility are needed.]

3. TECHNICAL ARCHITECTURE
--------------------------------------------------------------------------------

Q: Describe the high-level architecture (frontend, backend, database).
A: Frontend: React 19 + TypeScript + Vite, SPA. Backend: either Node.js (Express)
   or FastAPI (Python); same API (auth, scan upload/list, health). Database:
   MongoDB (users, scans metadata). Optional: separate uploads directory for
   DICOM files.

Q: How does the frontend communicate with the backend?
A: REST API over HTTP/HTTPS. Environment variable (e.g. VITE_API_URL) for API
   base URL. Auth: JWT in Authorization header (Bearer token). Key endpoints:
   POST /api/auth/login, GET /api/auth/me, POST /api/scan/upload, GET /api/scan,
   GET /api/scan/:id/file. [Reference: Study Materials/Frontend_Backend_Communication.txt]

Q: Why MongoDB?
A: [Prepare: flexible schema for scans (originalName, size, patientName, studyDate,
   modality, createdAt); easy to extend; you may have used it for users as well.
   Reference: Study Materials/MongoDB_Steps.txt, mongo setup 2.txt.]

Q: Why offer both Node and FastAPI backends?
A: Same API contract so the frontend works with either. Flexibility for the team
   (JavaScript vs Python), or future ML/AI integration in Python.

--------------------------------------------------------------------------------
4. SECURITY & AUTHENTICATION
--------------------------------------------------------------------------------

Q: How is the user authenticated?
A: Login returns a JWT. Frontend stores it (e.g. localStorage or memory) and sends
   it in the Authorization header for protected routes. ProtectedLayout (or
   similar) redirects unauthenticated users to login.

Q: How do you protect sensitive routes and data?
A: Backend validates JWT on protected endpoints (e.g. /api/scan, /api/auth/me).
   Frontend: protected routes wrapped in a layout that requires auth; no token →
   redirect to login.

Q: What about DICOM file storage and access?
A: Files stored in a configured upload directory; served via backend (e.g.
   GET /api/scan/:id/file) with access control (only authenticated users or
   scan owner). Reference: DICOM_Data_Confidentiality.txt.

--------------------------------------------------------------------------------
5. FRONTEND IMPLEMENTATION
--------------------------------------------------------------------------------

Q: Why React and TypeScript?
A: React: component-based UI, large ecosystem, good for dashboards and forms.
   TypeScript: type safety, fewer runtime errors, better IDE support and
   maintainability.

Q: What is Vite and why use it?
A: Vite is the build tool and dev server: fast HMR, native ES modules, optimized
   production builds. Replaces CRA for a modern React setup.

Q: How do you handle routing and layout?
A: React Router (BrowserRouter, Routes, Route). /app is protected; AppLayout
   provides sidebar, header, breadcrumbs, and Outlet for nested routes (Dashboard,
   patients, upload, analysis, reports, compare, analytics, etc.).

Q: How did you implement animations and interactivity?
A: Framer Motion for page transitions, staggered lists, hover/tap on cards and
   buttons. Global CSS for card/button/input hover and transitions. AnimatePresence
   for route changes and list item add/remove (e.g. upload list).

Q: How do you parse DICOM metadata in the browser?
A: dicom-parser library: parse DICOM buffer and read tags (e.g. patient name,
   study date, modality) for display or pre-fill before upload. Used in
   UploadScan (and possibly elsewhere).

--------------------------------------------------------------------------------
6. BACKEND & API
--------------------------------------------------------------------------------

Q: How does DICOM upload work end-to-end?
A: User selects/drops files → frontend validates .dcm/.dicom (and optionally
   extracts metadata with dicom-parser) → POST /api/scan/upload (multipart)
   → backend validates extension/MIME, size limit (e.g. 500 MB), stores file and
   metadata in DB → returns scan id/metadata.

Q: How do you prevent invalid or malicious uploads?
A: Backend: check file extension (.dcm, .dicom) and/or MIME type
   (application/dicom); enforce max file size; store files outside web root with
   controlled access. Optional: virus scan, validation of DICOM structure.

Q: What is stored in the database for a scan?
A: Typically: _id, originalName, size, patientName, studyDate, modality,
   createdAt, and path or reference to the stored file.
